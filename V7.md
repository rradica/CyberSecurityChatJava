## 1) ChatOrchestrator defensiver machen

Aktuell wird im Triage-Pfad sinngemäß mit `assessment.recommendedAction()` gearbeitet.

Ziel:
Die Workflow-Ausführung soll explizit auf der typisierten Validierung des DTO basieren.

Bitte ändere das so, dass:
- `handleTriageAssessment(...)` zuerst mit `assessment.hasValidAction()` arbeitet
- nur dann eine Aktion weiter geprüft wird
- keine stillschweigende Annahme mehr besteht, dass jede LlmService-Implementierung bereits sauber sanitizen wird

Wichtig:
- Die Logik soll klein und gut lesbar bleiben
- Kein großer Umbau
- Einfach nur etwas defensiver und expliziter

---

## 2) `riskLevel` in `TriageAssessment` validieren

Aktuell wird `recommendedAction` bereinigt und `confidence` geklemmt.
Bitte ergänze auch eine Validierung von `riskLevel`.

Erlaubte Werte:
- low
- medium
- high
- critical

Wenn ein anderer Wert kommt:
- auf einen sicheren Default setzen, vorzugsweise `medium`

Wichtig:
- Bitte einfach halten
- Keine unnötigen Enums, wenn der Code dadurch komplizierter wird
- Wenn ein kleines `Set<String>` im DTO reicht, ist das gut

Optional:
- Eine kleine Hilfsmethode wie `hasValidRiskLevel()` ist okay, wenn sie den Code lesbarer macht

---

## 3) Real-LLM-Smoke-Test vorbereiten

Ich möchte für den echten LLM-Pfad eine kleine, einfache Verifikationsmöglichkeit haben.

Bitte implementiere eine kleine, pragmatische Lösung, die zeigt, ob der echte LLM-Pfad mit `OpenAiChatService.assessTriage(...)` grundsätzlich funktioniert.

Bevorzugte Optionen:
- ein kleiner Test auf Service-Ebene
- oder ein sehr kleiner interner Smoke-Test-Service
- oder ein einfacher Testfall, der `assessTriage()` mit einem festen Prompt aufruft

Wichtig:
- Kein schwerer End-to-End-Monstertest
- Kein aufwendiges Testframework-Konstrukt
- So wenig Komplexität wie möglich

Der Smoke-Test soll mindestens prüfen:
- `assessTriage()` liefert ein nicht-null `TriageAssessment`
- `summary` ist gesetzt
- `riskLevel` ist nach `sanitized()` gültig
- `recommendedAction` ist entweder gültig oder null
- bei Fehlern greift der sichere Fallback

Wenn echte API-Calls im automatischen Testlauf problematisch sind:
- bitte den Smoke-Test so gestalten, dass er nur bei gesetztem API-Key bzw. explizit aktiviertem Profil läuft
- ansonsten soll der normale Testlauf stabil bleiben

Wichtig:
Ich möchte nicht, dass CI/CD oder lokales Entwickeln ohne OpenAI-Key kaputtgeht.
