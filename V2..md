Bitte überarbeite die bestehenden Workshop-Schwachstellen in unserer Spring-Boot-/Spring-AI-Anwendung gezielt, damit sie realistischer, wirksamer und besser zur Workshop-Beschreibung passen.

Wichtiger Kontext:
Die Anwendung ist bewusst eine Red-Team-vs-Blue-Team-Workshop-App zum Thema AI Security mit RAG und Tools.
Die Schwachstellen müssen absichtlich im Code vorhanden sein.
Sie dürfen NICHT wegoptimiert werden.
Sie müssen:
- realistisch wirken
- nicht trivial sein
- reproduzierbar auslösbar sein
- im normalen Anwendungscode leben
- für Teilnehmer gut verständlich und fixbar bleiben
- zur Workshop-Ankündigung passen:
  - untrusted content
  - RAG-/Kontextverarbeitung
  - Tool-/Workflow-Aktionen
  - ein plausibler Use Case, der kippt
  - Prompting ist keine Security Boundary

Technischer Rahmen:
- Spring Boot
- Spring AI
- einfacher, lesbarer Code
- browser-/Codespaces-tauglich
- keine unnötige Komplexität
- kleine Klassen und Methoden
- keine neue Infrastruktur
- keine Vector-DB
- keine externe Datenbank, wenn nicht nötig

Ganz wichtige Architekturregel:
**Berechtigung vor Kontext**
Das heißt:
1. Actor bestimmen
2. Zweck / Aktion bestimmen
3. erlaubte Quellen bestimmen
4. Kontext aufbauen
5. erst dann LLM / Tool-Vorschlag

Wichtig:
- Das LLM darf nicht über Berechtigung entscheiden.
- Das LLM darf nicht über Quellenfreigabe entscheiden.
- Das LLM darf nicht über Tool-Autorisierung entscheiden.
- Die sicherheitskritischen Schwächen sollen app-seitig entstehen.

Bitte prüfe und verbessere gezielt die folgenden Bugs:

## 1) BUG_TRUST_MERGE verbessern

Ist-Zustand:
Aktuell wird im PromptBuilder bei aktivem BUG_TRUST_MERGE im Wesentlichen nur das Trust-Level der Quellen nicht mehr angezeigt.
Das ist als Bug-Idee okay, aber noch zu schwach und zu stark vom Modellverhalten abhängig.

Ziel:
Mache BUG_TRUST_MERGE realistischer und wirksamer.

Anforderungen:
- Trusted und untrusted Inhalte sollen im Bug-Fall nicht nur ohne Label erscheinen, sondern strukturell zu stark miteinander verschmelzen.
- Der Bug soll wie ein plausibler Produktfehler wirken:
  - "helpfulness"
  - "single consolidated guidance view"
  - "weniger kognitive Last"
- Im Bug-Fall sollen untrusted Supplier- oder Evidence-Quellen in einem gemeinsamen Guidance-Block mit internen Policies landen.
- Konflikte zwischen interner Policy und externer/untrusted Evidenz sollen im Bug-Fall nicht explizit markiert werden.
- Im Fix-Fall sollen Quellen klar getrennt und sichtbar markiert werden, z. B.:
  - internal policy
  - internal runbook
  - external evidence
  - untrusted supplier note
- Der Code soll einfach lesbar bleiben.

Bitte ändere also nicht nur Labels, sondern die Struktur der Kontextaufbereitung so, dass der Bug realistischer und wirksamer wird.

## 2) BUG_THREAD_STICKINESS wirksam machen

Ist-Zustand:
Aktuell werden Konversationshistorie und letzter Retrieval-Kontext in der Session gespeichert, aber der Sticky-Kontext hat noch zu wenig echte Wirkung.
Dadurch ist der Bug eher theoretisch als praktisch ausnutzbar.

Ziel:
Mache BUG_THREAD_STICKINESS zu einer realen, ausnutzbaren Schwachstelle.

Anforderungen:
- Wenn BUG_THREAD_STICKINESS aktiv ist, soll alter Session-/Conversation-Kontext in mindestens einem sicherheitsrelevanten Pfad tatsächlich wiederverwendet werden.
- Das soll realistisch wirken, z. B. als:
  - Performance-Optimierung
  - "reuse prior case context"
  - "avoid redundant retrieval"
- Geeignete Pfade wären insbesondere:
  - handover
  - workflow
  - similar cases
- Beispiel:
  - Ein Analyst öffnet einen Fall und lädt sensitiven Kontext.
  - Danach wechselt die Rolle auf contractor.
  - Bei aktivem Bug wird der alte Kontext ganz oder teilweise weiterverwendet.
- Der Bug soll nicht wie eine plumpe Hintertür wirken, sondern wie ein Session-/State-Fehler in einem Chat-/Case-System.
- Im Fix-Fall muss der Kontext sauber an Actor und aktuellen Request gebunden werden.

Bitte implementiere den Bug so, dass er wirklich Auswirkungen auf Retrieval, Prompt oder Tool-Entscheidungen hat.

## 3) BUG_TOOL_FASTTRACK realistischer und zentral machen

Ist-Zustand:
Der bisherige Tool-/Workflow-Pfad wirkt noch etwas zu direkt und demohaft.
Wir brauchen einen realistischeren Incident-Case.

Wichtiger fachlicher Zielcase:
Nicht direkte "Allowlist"-Freigabe, sondern ein realistischer Triage-/Workflow-Fehler.

Bevorzugter Incident-Case:
- Eine manipulierte, aber plausibel wirkende Quelle beeinflusst die Bewertung eines Falls.
- Dadurch wird ein legitimer Workflow-Schritt zu leicht ausgelöst.
- Die Aktion soll eher so aussehen:
  - mark_case_likely_false_positive
  - set_case_priority_low
  - route_case_to_finance_queue
  - attach_supplier_trust_note
- Der eigentliche Incident entsteht im zweiten Schritt:
  - Fall wird falsch behandelt
  - Security-Eskalation bleibt aus oder wird verzögert
  - ein Folgefall rutscht später leichter durch

Anforderungen:
- Die Tool-/Workflow-Entscheidung darf nicht einfach nur vom vom Nutzer genannten Action-String abhängen.
- Das Modell oder die App soll zunächst eine Empfehlung / Triage-Einschätzung erzeugen.
- Danach soll app-seitig geprüft werden, ob daraus ein Workflow-Schritt entsteht.
- Im Bug-Fall soll die Tool-Policy zu schwach auf:
  - Trust-Level
  - Quellenherkunft
  - Rollenbindung
  - Evidenzqualität
  reagieren.
- Im Fix-Fall soll genau diese Policy-Härtung möglich und einfach sichtbar sein.
- Der Bug muss reproduzierbar sein, ohne auf verrücktes Modellverhalten angewiesen zu sein.

Wichtig:
Der Case muss zur Workshop-Ankündigung passen:
"Eine manipulierte Quelle beeinflusst das Modell so, dass es eine Aktion anstösst – und aus einer scheinbar normalen Interaktion wird ein echter Security Incident."

Bitte setze das so um, dass:
- untrusted content
- Retrieval / Kontext
- Tool-/Workflow-Aktion
- Incident-Effekt
sauber zusammenhängen.

## 4) BUG_EXISTENCE_ORACLE prüfen und ggf. stärken

Bitte prüfe außerdem, ob unser Similar-Cases- oder Evidence-Pfad bereits stark genug einen realistischen Existence-Oracle darstellt.

Wenn nicht, verbessere ihn so, dass:
- sensible Vorfälle nicht direkt vollständig offengelegt werden,
- aber ihre Existenz, Anzahl, Titel, Tags oder Art indirekt erkennbar wird,
- und das für das Red Team als Reconnaissance nützlich ist.

Auch hier:
- realistisch
- nicht cartoonhaft
- durch Blue Team gut fixbar

## 5) Qualität und Einfachheit erhalten

Bei allen Änderungen gilt:
- kein Overengineering
- keine unnötigen neuen Schichten
- keine komplizierten Patterns
- keine unnötige Magie
- Code für Workshop-Teilnehmer schnell lesbar halten
- kleine Methoden
- gute Namen
- einfache Kommentare
- Bugs nicht so offensichtlich kommentieren, dass sie sofort wie CTF-Lösungen wirken

Bitte reduziere überdeutliche "BUG:"-Kommentare im Produktionscode.
Stattdessen:
- Code selbst soll den Fehler plausibel zeigen
- Kommentare eher neutral formulieren
- falls nötig, zusätzliche Erklärung lieber in Tests oder README

## 6) Erwartete Lieferung

Bitte liefere:
1. konkrete Code-Änderungen für die relevanten Klassen
2. neue oder angepasste Methoden
3. kurze Begründung je Bug, warum die neue Version realistischer ist
4. falls nötig neue kleine Hilfsklassen oder DTOs
5. passende Tests oder Testanpassungen
6. keine unnötigen Änderungen außerhalb dieses Ziels

Bitte arbeite konkret am vorhandenen Code und verbessere diese Bugs gezielt, statt die Anwendung allgemein "sicherer" zu machen.
Die vorbereiteten Schwachstellen müssen erhalten bleiben – nur realistischer und wirksamer werden.
