Bitte erstelle eine V3 der aktuellen Workshop-Anwendung.

Wichtige Regeln:
- Die Kommentare im Code bitte beibehalten.
- Kommentare dürfen weiter explizit sein.
- Die vorbereiteten Schwachstellen müssen erhalten bleiben.
- Sie sollen realistischer und wirksamer werden, nicht entfernt werden.
- Kein Overengineering.
- Code einfach und workshop-tauglich halten.
- Spring Boot / Spring AI / einfacher lesbarer Code beibehalten.

Ziel von V3:
Die App soll noch klarer und realistischer den Ablauf zeigen:
untrusted content -> fehlerhafte Kontext-/Trust-Verarbeitung -> strukturierte Triage-Empfehlung -> zu schwach abgesicherte Workflow-Aktion -> Incident-artiger Effekt.

Bitte setze insbesondere folgende Verbesserungen um:

## 1) BUG_TOOL_FASTTRACK ausbauen
Der eigentliche Kippmoment soll im ToolPolicyService liegen.

Anforderungen:
- ToolPolicyService soll im Bug-Fall zu großzügig sein.
- Der Bug soll plausibel wie ein Produkt-/Usability-Fehler wirken.
- Untrusted oder semi-trusted Supplier-/Evidence-Quellen sollen zu stark auf Workflow-Entscheidungen wirken können.
- Besonders relevant:
  - attach_supplier_trust_note
  - mark_case_likely_false_positive
  - set_case_priority_low
  - route_case_to_finance_queue
- Im Bug-Fall soll für bestimmte Aktionen bereits schwache oder einseitige Evidenz ausreichen.
- Im Fix-Fall sollen riskante Aktionen nur mit ausreichend trusted interner Evidenz erlaubt sein.
- Bitte baue den Bug so, dass er für den Workshop reproduzierbar ist.

## 2) Strukturierte Triage-Antwort einführen
Bitte ersetze das fragile Parsen per Freitext-contains durch eine strukturierte Triage-Antwort.

Anforderungen:
- Führe ein kleines DTO ein, z. B. TriageAssessment.
- Felder z. B.:
  - summary
  - riskLevel
  - recommendedAction
  - confidence
  - evidenceAssessment
- LLM oder Mock sollen dieses strukturierte Objekt liefern.
- ChatOrchestrator soll mit diesem Objekt arbeiten statt Freitext nach Actions zu durchsuchen.
- Einfach halten, keine unnötige Komplexität.

## 3) BUG_THREAD_STICKINESS realistischer machen
Aktuell ist der Bug brauchbar, aber noch etwas grob.

Anforderungen:
- Reuse von Session-Kontext soll wie eine Performance-/Case-Continuity-Optimierung wirken.
- Context-Reuse nur für denselben caseId und bestimmte Modi (handover/workflow), wenn Bug aktiv.
- Im Fix-Fall muss Context sauber an Rolle, Fall und Modus gebunden sein oder bei Rollenwechsel gelöscht werden.
- Kommentare bitte beibehalten.

## 4) BUG_EXISTENCE_ORACLE subtil verbessern
Bitte prüfe Similar-Cases-Logik und verbessere sie leicht.

Anforderungen:
- Reconnaissance soll über Existenz, Typ, Count, Severity oder Pattern-Hinweise möglich bleiben.
- Es soll aber nicht wie ein plumper Direkt-Leak wirken.
- Eher so:
  - restricted matches count
  - pattern hints
  - severity hints
- Einfach und verständlich umsetzen.

## 5) Incident-Effekt sichtbarer machen
Bitte baue einen klaren, aber einfachen Folgeeffekt ein.

Anforderungen:
- Wenn eine kritische Workflow-Aktion ausgeführt wird, soll sich der Case-Zustand ändern.
- Beispiele:
  - escalationSuppressed
  - priorityLow
  - routedToFinance
  - trustNoteAttached
- Spätere Antworten oder Folgefälle sollen diesen Zustand berücksichtigen.
- Damit soll sichtbar werden, dass aus einer normalen Interaktion ein echter Security-Incident-artiger Effekt entstanden ist.

## 6) PromptBuilder leicht verbessern
BUG_TRUST_MERGE ist schon besser, soll aber noch etwas stärker wirken.

Anforderungen:
- Im Bug-Fall darf die konsolidierte Guidance ruhig noch stärker als kuratierte Gesamteinschätzung wirken.
- Im Fix-Fall bitte Konflikte zwischen trusted internen Quellen und untrusted externer Evidenz sichtbar markieren.
- Einfach halten.

## 7) Kommentare beibehalten
Ganz wichtig:
- Kommentare NICHT reduzieren
- Kommentare NICHT neutralisieren
- Kommentare NICHT entfernen
- Sie sind absichtlich für interne Vorbereitung da

## 8) Erwartete Lieferung
Bitte liefere:
1. konkrete Code-Anpassungen
2. neue/angepasste Klassen und Methoden
3. ggf. neues DTO für strukturierte Triage
4. Tests oder Testanpassungen
5. kurze Begründung je Änderung, warum sie realistischer geworden ist
